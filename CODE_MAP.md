## ⚠️ Read this first: this map is a living, imperfect tool

This map describes a **living codebase**. Treat it as a fast-but-fallible index, never as
ground truth. Concretely:

1. **The file is the source of truth, then this map, then the guide — in that order.**
   The map was written while looking at the code, so where the **bootstrap guide**
   disagrees with the map on a factual detail, trust the map. Where the **map** disagrees
   with what you find when you actually grep and read the code, trust the code and consider
   the map stale at that spot. (Known drift already found: the guide says save-migration
   head is **v8**; the code is actually at **v11**. See *Map-vs-guide conflicts* below.)

2. **Anchors can rot.** Every anchor grep-matched when this map was built. As soon as
   someone edits a class signature, renames a method, or reflows a comment, the
   corresponding anchor may stop matching. If an anchor returns nothing:
   - Grep a **shorter substring** of it (drop trailing args/braces).
   - Grep a **nearby unique token** from the entry's description.
   - Search the parent class/section and scan locally.
   An anchor that misses is a signal the code moved, not that the concept is gone.

3. **Coverage is good but not total.** The crawl indexed the whole file, but some dense
   regions (long draw calls, CSS blocks, base64/data-URI noise) were deliberately
   summarized or skipped — see `(skipped: …)` notes inline. Absence from this map does
   **not** mean absence from the code. When in doubt, grep the file directly.

4. **Descriptions are compressed.** Each `Does:` line is a lossy summary. Before you edit,
   open the actual code around the anchor and read ~150–400 lines. The map gets you to the
   right neighborhood; it does not replace reading.

5. **This map is append-friendly but not auto-updated.** If you add a system, add an entry
   (match the existing format: a backtick anchor that literally appears in the file, a
   one-line `What`, a `Does`, and `Refs` where obvious). If you rename something, fix its
   anchor here. A five-minute map update saves the next reader an hour.

### Recommended workflow for a future AI working in this codebase

- **Start from intent, land via anchor.** Read the relevant guide section for the *why*,
  find the matching anchor here for the *where*, then `grep -n 'anchor' index.html` and read
  locally. Do **not** ingest the whole file — it will blow your context and teach you
  nothing you couldn't get locally.
- **Adding content = adding a registry entry.** Most features are data-driven (see the
  *Registries* section). Before writing imperative code, check whether a registry already
  owns that category and add one entry instead. `ARMORY_UPGRADES` is the exemplar pattern.
- **Route all Intel rewards through `RunLedger.addIntel(`** — never write `saveData.intel`
  directly. Reset run-scoped state in `startNewGame(`. Migrate saves additively (add a new
  `{ to: N }` step; never edit an old one).
- **Respect the regression-sensitive zones** (listed below) — a handful of choke points
  ripple across the whole game. Handle with tongs and re-validate the JS (`node --check`
  on the extracted `<script>`) before shipping.
- **When you finish a change, update this map** for anything you moved, renamed, or added.

## Regression-sensitive zones (handle with tongs)

Per guide §7 — editing near these ripples across the whole game. Grep, read widely, and
re-validate before shipping.

- **The `dt` scaler** (time dilation / Enraged boss trait / Adrenaline). Everything
  time-based keys off `dt`; scaling it touches physics, animation, cooldowns, and reload at
  once. Grep: `updatePhysics(`, and the boss-trait speedups in the `class Boss` subclasses.
- **`takeDamage(amt, pushDir, pushForce)`** — the single damage choke point. Player-damage
  hook drives RunLedger, objective interrupts, clean-wave tracking, Blood Wager fail. Both
  `class Entity` and `class Player` define it (Player overrides).
- **`addIntel(`** (`RunLedger.addIntel`) — the single Intel write path. Honors
  eligibility/taint. Route *all* rewards through it.
- **Gamepad dispatch** — `pollGamepad` fans out to per-screen handlers gated by state flags
  and `dpadCooldown`. New overlays need a handler *and* a dispatch entry. Grep the
  `handle*MenuNavigation` family and `handleChaosDraftMenuNavigation` (checked before the
  pause/state branches).
- **`draw()` early-returns in `start` state** — the world canvas is NOT live on the menu.
  Menu-time visual effects must be DOM/CSS (see the prestige FX CSS), not world particles.
- **`startNewGame()`** — miss a reset here and run-scoped state (e.g. `chaosDraftState`,
  `runOverclocks`) leaks across runs.
- **The known wart:** `maxStamina !== undefined` is used as the "is this the Player" check
  in base-class loops (guide §5). Still live in the code. A future `Entity` subclass that
  defines `maxStamina` would silently inherit player-only logic. Disarm path if it ever
  bites: add explicit `this.isPlayer = true` in the `Player` constructor and batch-replace.
